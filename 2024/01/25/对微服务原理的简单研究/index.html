对微服务原理的简单研究

摘 要：随着信息技术的不断发展，软件系统的规模和复杂性不断增加，传统的单体应用架构难以满足需求。因此，能够将将整个应用拆分为小型服务，每个服务专注于一个特定的业务功能的微服务架构应运而生。与此同时，随着物联网、边缘计算和大数据应用的快速发展，传统的中心化云服务面临着一系列挑战。为了满足日益增长的实时数据处理需求、降低延迟、提高应用的灵活性和可靠性，云边协同的微服务架构成为一种不可或缺的解决方案。本论文将探讨微服务架构的原理、优点以及基于微服务架构的云边协同的优势。

关键词： 1；微服务架构 2；云边协同 3；端云一体化 4；去中心化

1 微服务架构原理及特点

基于微服务架构形高内聚、低耦合的系统，降低系统的迭代成本，保障系统的弹性扩展能力。服务之间通过服务中心进行注册、访问。[1]下以基于微服务的用户体验评价平台构建为例，解释微服务平台的服务拆分，自治性，去中心化数据管理和分布式架构的特点。基于微服务的用户体验评价平台构建图解见Fig.1.

Fig.1

1.1 服务拆分

微服务架构的核心在于将整个应用拆分为小型服务，每个服务专注于一个特定的业务功能。微服务框架将原先传统的单体应用框架中的应用服务进行了拆分，如Fig.1中的调研管理、用户体验评价、数据分析三个服务，服务拆分是后面几个特点的来源。

1.2 自治性

拆分后的每一个微服务是自治的，每个服务都有自己的数据库和业务逻辑。Fig.1中的调研管理下问卷编制等业务与用户评价体系和数据分析下的业务进行了分隔。这种自治性使得微服务可以独立开发、测试、部署和扩展，降低了服务之间的耦合度，提升了每个服务的可拓展性。

1.3 去中心化数据管理

每个微服务都有自己的数据库，避免了传统共享数据库的问题。在Fig.1所示的基于微服务的用户体验评价平台构建中的数据层便将数据以TP型和AP型进行划分，在应用服务调用数据时根据说需要的目标指向性的调用。去中心化的数据管理降低了数据存储服务器的传输压力，也更利于维护和保障数据的稳定。

1.4 分布式架构（和1.1没什么区别，纯水）

微服务的核心思想是将一个庞大的单体应用拆分为一系列小的、可独立部署的服务。而这些独立部署的服务的调用和服务之间的互相调用就需要通过网络协议（常见的有http传输协议等）来进行。

1.5 注册中心、微服务网关和api网关

（这个是比如小程序之类的微服务架构的大体结构，了解不亏）

同一个应用调用多个不同的微服务或者不同的应用调用一个微服务时需要通过网络协议来进行。微服务之间的互相调用也需要通过网络协议。（人话：分布式导致联络需要通过网络）微服务的注册中心是微服务架构中的一个关键组件，用于帮助微服务发现和管理[3]（提供互相调用的平台，类似于交换货物的市场）。其主要功能是维护正在运行的微服务实例的信息，以便其他服务能够发现和与其通信。微服务在注册中心注册并将自身业务接口（API）[2]提供给注册中心。注册中心提供了一种集中式的机制，使得微服务之间的通信和协作更为灵活和动态。微服务网关和 API 网关是在微服务架构中用于管理和提供对微服务的访问的关键组件。微服务网关是一个专门的服务器，用于处理微服务之间的通信。它提供了一个入口点，将客户端请求路由到适当的微服务实例。API 网关更侧重于整个系统的 API 管理和暴露。它是一个集中的入口点，用于处理所有与客户端的通信，并为客户端提供统一的 API。[4]

1.6 微服务的调用

那么软件和微服务之间以及微服务与微服务之间是如何调用的呢？

（1） 对于微服务之间的互相调用：注册中心给接口

在注册中心注册后的微服务能够从注册中心中得到目标微服务（同在注册中心注册并向注册中心提供了自身API的微服务）的接口并发起点对点的具体调用（也就是说具体的API调用不需要再经过注册中心）。这时，查找微服务目标地址（向注册中心发送请求）的数据流与微服务接口调用的数据流（具体调用API）实现了彻底分离。

（2） 应用调用微服务：

不同于微服务之间的API调用，前端应用需要调用微服务直接暴露的API，由于前后端的分离，前端应用一般无法直接声明并调用微服务的API。这时会引入微服务网关和API网关。（人话：技术多样性导致很多函数方法之间不能互相调用，比如一个用c开发的前端不能直接操作java开发的后端；前后端分离导致前后端联系困难）

方式一：APP首先调用了微服务网关的接口，向微服务网关发送了目标微服务的地址连接，微服务网关在拿着这个地址链接前往注册中心寻址，找到具体的调用地址后通过反向代理返回给app进行调用。这里微服务网关起到了负载均衡的作用。Fig.2

方式二：微服务通过直接暴露或通过微服务网关间接将API暴露给API网关，API网关具体管控每一个API。App直接通过反向代理向API网关申请调用。这种方式在跨应用的交互中有很大优势。具体结构可参考Fig.3[5]

2 微服务架构的优点

2.1 灵活性和可扩展性

分布式部署允许每个微服务独立运行，可以根据需求进行水平扩展。这种独立伸缩性使得系统能够更好地适应不同的负载和需求。团队可以选择在需要的时候扩展特定服务，而不是整体扩展整个应用。这种按需扩展的方式更加经济高效。服务量较少的微服务可以减少服务器的部署，或使用成本更低的低算力服务器。同时可以针对高峰时段通过及时的增加服务器数量等方式满足需求，在淡季又能够关闭部分服务器以节约成本。

2.2 团队自治和技术多样性

自治性有助于提高开发速度和灵活性。每个微服务可以由一个小团队开发和维护，服务之间的耦合度低也就意味着与整体而言，不相关的模块或功能的开发可以更加高效便捷。由于每个服务都是独立的，开发团队可以选择不同的编程语言、框架和技术栈来实现各个服务。这种灵活性使得团队能够更好地根据具体服务的需求选择最适合的工具，而不受整体系统的限制。

2.3 容错性

此外，微服务的独立性也带来了更容易维护的优势。当系统中的一个服务需要更新或修复时，只需关注该服务而无需影响到整个应用。这降低了维护的复杂性，同时也提高了系统的可靠性。

同时，微服务框架的去中心化结构（包括去中心化的数据管理和去中心化的微服务之间互相的调用，见1.6微服务的调用）

2.4 快速部署

微服务的独立性还体现在独立部署和版本控制方面。每个服务都可以独立部署，不影响其他服务的运行。这意味着团队可以更频繁地发布新的功能、修复bug或进行性能优化，而无需等待整个应用的发布周期。

版本控制也变得更为灵活。不同的服务可以在不同的版本上运行，这使得系统升级更加平滑。同时，通过采用适当的技术手段，如容器化[6]和持续集成/持续交付（CI/CD）[7]，可以实现自动化的部署流程，进一步提高开发和部署的效率。

3 基于微服务的端云一体化

在传统云计算中，数据通常会被发送到远程数据中心进行处理和存储，这可能导致延迟和带宽问题，特别是对于需要即时响应的应用程序和服务来说，这种延迟是不可接受的。端云一体通过在数据产生的地方、即“边缘”（Edge）或“端”（Edge）上部署计算资源，实现数据处理和存储，从而解决了这一问题。

端云一体（Edge Computing）是一种新兴的计算模型，是一种云计算模型的扩展，它涉及将数据处理和存储功能尽可能地移到离数据生成源头更近的位置，将云计算的能力扩展到网络边缘，为许多需要低延迟、高可靠性和快速响应的应用场景提供了更为合适的解决方案。

终端到云集成可重构物联网（Terminal to Cloud Integrated Reconfigurable Internet of Things,TCIR-Io T）利用可重构技术进行物联网终端嵌入式软件的开发，对平台各个功能模块进行封装。同时，模块化平台功能采用微服务技术搭建平台的分布式服务架构，降低开发的成本，可提高平台运行的稳定性和可靠性。因此，在TCIR-Io T应用层软件设计与实现中，运用物联网技术、可重构技术、微服务技术等搭建平台的架构，开发应用层软件的注册中心、分布式文件中心、动态负载均衡器以及后台管理平台等，实现可重构物联网系统平台的端云一体化。
